## Overview

The MVP uses the Android app as a local offline queue and the Spring Boot backend as a thin Markdown writer for the Obsidian Vault. The backend owns no AI logic and stores only enough metadata to prevent duplicate files.

## Backend

Expose `POST /api/captures/notes` with a JSON body containing:

- `id`: stable UUID generated by Android
- `content`: original note body
- `createdAt`: capture timestamp
- `source`: source label, defaulting to the Android app

The backend validates that `id` and `content` are present, writes a Markdown file to `pkms.vault.inbox-path`, and returns the existing result if the same ID is posted again. Each Markdown file contains YAML frontmatter with the capture ID, source, creation timestamp, sync timestamp, and sync status.

File naming uses a timestamp plus a short ID suffix to keep files readable while avoiding collisions. A small local metadata file under `pkms.sync.index-path` maps capture IDs to written file paths.

## Android

The Android app contains:

- A Room `captures` table for notes.
- A repository that saves notes as `PENDING`.
- A WorkManager worker that posts pending notes to the backend and marks them `SYNCED` on success.
- Connectivity-constrained one-time sync work scheduled after each note save and at app start.
- A Compose UI with a note text field, save button, and visible pending/synced counts.

The backend base URL is a build config value so a local laptop IP can be configured later without changing app logic. The default points at the Android emulator host address.

## Error Handling

Android keeps failed sync attempts pending and lets WorkManager retry. The backend rejects blank content, malformed timestamps, and unsafe/missing IDs with client errors instead of writing files.

## Configuration

Backend defaults write to `./vault/Inbox` and keep sync metadata in `./data/synced-captures.properties`, both configurable in `application.properties`.
